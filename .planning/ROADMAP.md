# Roadmap: OtakuPulse Stabilization & Optimization

## Overview

Three phases harden OtakuPulse from a working app into a reliable one. Phase 1 eliminates crashes and data-correctness bugs that undermine every other layer. Phase 2 adds lifecycle safety and security hardening so the app shuts down cleanly and handles secrets correctly. Phase 3 locks in performance gains and builds a test suite that asserts against the now-stable behavior.

## Phase Details

### Phase 1: Foundation Correctness
**Goal**: The app starts reliably, dedup produces correct results for Japanese titles and URL variants, the DB supports concurrent reads, and the CI builds without a live database.
**Depends on**: Nothing (first phase)
**Requirements**: SAFE-01, SAFE-02, SAFE-03, SAFE-04, BUG-01, BUG-02, BUG-03, BUG-04, PERF-01, DEP-01, DEP-02, DEP-03, FRNT-01
**Success Criteria** (what must be TRUE):
  1. App launches successfully even when the DB init fails — a human-readable error message appears instead of a silent crash
  2. Feed articles with reordered URL query parameters (e.g., `?a=1&b=2` vs `?b=2&a=1`) are recognized as duplicates
  3. Japanese titles using half-width katakana and their full-width equivalents collapse into a single deduplicated article
  4. DeepDive Q&A answers update when the underlying article summary changes rather than returning stale cached answers
  5. The CI build completes without a live SQLite database present (sqlx offline mode)
**Plans**: 01-01 (Safety & Infrastructure), 01-02 (Data Correctness), 01-03 (Dependencies & Frontend)

### Phase 2: Resilience & Security
**Goal**: The scheduler shuts down cleanly without risking DB corruption, Settings changes take effect immediately, the app serves cached content when offline, and no API key leaks into logs.
**Depends on**: Phase 1
**Requirements**: RESL-01, RESL-02, RESL-03, RESL-04, RESL-05, SEC-01, SEC-02, SEC-03
**Success Criteria** (what must be TRUE):
  1. Closing the app window flushes in-flight DB writes and exits within 5 seconds — no process hang, no DB corruption
  2. Changing the collection interval in Settings takes effect on the running scheduler without restarting the app
  3. When all external APIs are unreachable, the Feed shows articles from the last 72 hours rather than an empty screen
  4. A Perplexity API key cannot be reconstructed from the application log files
  5. Broken RSS feeds surface a visible error in Settings rather than silently disappearing from the feed
**Plans**: 02-01 (Scheduler Lifecycle — Shutdown + Hot-Reload), 02-02 (Resilience — Error Visibility + Offline Mode), 02-03 (Security Hardening)

### Phase 3: Performance & Test Coverage
**Goal**: The digest generates in parallel, DB round-trips are minimized, FTS search does not load all matches into memory, and a comprehensive test suite guards all the Phase 1–2 fixes.
**Depends on**: Phase 2
**Requirements**: PERF-02, PERF-03, PERF-04, PERF-05, PERF-06, TEST-01, TEST-02, TEST-03, TEST-04, TEST-05, TEST-06, TEST-07
**Success Criteria** (what must be TRUE):
  1. Digest generation for all four categories completes in roughly one-quarter of the previous time (parallel tokio::join! vs. serial loop)
  2. The highlights section loads with a single DB query instead of one query per article
  3. A dedup test suite with 20+ cases passes, covering Unicode normalization, URL canonicalization, and content hash edge cases
  4. The TypeScript hook error-handling tests demonstrate that invoke errors surface as typed AppError shapes, not generic Error instances
  5. Coverage reports are generated by `cargo-llvm-cov` and `@vitest/coverage-v8` and show the test run results
**Plans**: 03-01 (Rust Performance Optimizations), 03-02 (Dedup & Rate Limiter Tests), 03-03 (Scheduler & Scoring Tests), 03-04 (TS Tests & Coverage Infrastructure)
**UI hint**: yes
