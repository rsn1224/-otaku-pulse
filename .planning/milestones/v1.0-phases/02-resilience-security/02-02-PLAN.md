**collector.rs changes:**
1. Define a new struct at the top of the file:
   ```rust
   #[derive(Debug, Clone, serde::Serialize)]
   pub struct FeedError {
       pub feed_id: i64,
       pub feed_name: String,
       pub error_message: String,
       pub consecutive_errors: i64,
   }
   ```
2. Change `refresh_all()` return type from `Result<u32, AppError>` to `Result<(u32, u32, Vec<FeedError>), AppError>` where the tuple is `(articles_saved, feeds_processed, errors)`.
3. Inside `refresh_all()`, collect per-feed errors into a `Vec<FeedError>` instead of only logging them. When a feed fetch/parse fails:
   - Still call `feed_queries::update_feed_failure()` (existing behavior)
   - Also push `FeedError { feed_id, feed_name, error_message: e.to_string(), consecutive_errors }` into the errors vec
4. Return `Ok((total_saved, feeds_processed, errors))` at the end.

**scheduler.rs changes (collect_loop call site):**
1. Update the `refresh_all()` call site in `collect_loop` to destructure `(saved, _processed, errors)`
2. Include error count in the `collect-completed` event payload:
   ```rust
   app_handle.emit("collect-completed", serde_json::json!({
       "fetched": saved,
       "errors": errors.len()

   ```
3. If `errors.len() > 0 && saved == 0`, also emit `collect-failed` event:
   ```rust

       "message": "All feeds failed to fetch",
       "errorCount": errors.len()
   ```

**RESL-04 — deepdive_service.rs (or discover_ai.rs commands):**
1. In the DeepDive question handler (likely `ask_deepdive` command or its service function):
   - At session start (first question): snapshot the current provider by reading `app_state.llm.read()?.provider` and store it alongside the conversation (in the DB `deepdive_questions` table as a new `provider` TEXT column, or in the command's session state)
   - On subsequent questions: read current `app_state.llm.read()?.provider` and compare with the stored provider
   - If mismatch: return `AppError::InvalidInput("LLM provider changed since conversation started. Please start a new conversation.".to_string())`
   - If match: proceed normally
2. If a migration is needed for the `provider` column in `deepdive_questions`:
   - Create `src-tauri/migrations/009_phase2_resilience.sql` with `ALTER TABLE deepdive_questions ADD COLUMN provider TEXT;`
   - Existing rows with NULL provider are treated as "any provider" (no validation for legacy data)

**RESL-05 — Frontend offline mode:**
1. In `src/stores/useSchedulerStore.ts`, add state:
   ```typescript
   isOffline: boolean;
   setOffline: (offline: boolean) => void;
   ```
   Initialize `isOffline: false`.
2. In `src/App.tsx` event listeners section, add listeners for:
   - `collect-failed`: `useSchedulerStore.getState().setOffline(true)`
   - `collect-completed`: `useSchedulerStore.getState().setOffline(false)` (auto-recovery)
3. In the Feed/Discover view component (likely `src/components/discover/` area), add a conditional banner:
   ```tsx
   {isOffline && (
     <div className="bg-amber-900/50 text-amber-200 px-4 py-2 text-sm text-center">
       Network unavailable — showing cached articles
     </div>
   )}
   ```
4. No backend changes needed for cached content display — `get_discover_feed` already reads from DB. The `created_at >= datetime('now', '-72 hours')` filter is already in effect because the discover query sorts by recency and only shows unread articles.

**Note on RESL-05 simplicity:** The app already shows DB-cached content. "Offline mode" is primarily about:
(a) Not showing an error when collection fails due to network (covered by collect-failed event)
(b) Showing a visual indicator that content may be stale (covered by the banner)
(c) Auto-recovering when network returns (covered by collect-completed clearing the flag)
