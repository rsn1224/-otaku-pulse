# Phase 4 — UI Design Contract

> Visual and interaction contract for Phase 4: Design Token Foundation.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Scope note:** Phase 4 is a pure CSS/token layer — no component JSX is modified.
> This contract defines the token vocabulary that Phase 5 components will consume.
> All HEX values for surface layers and accent colors are marked [STITCH-VALUE] and
> MUST be confirmed via the Stitch mockup session (D-01/D-02) before being committed.

## Spacing Scale

**Source:** design.md §4 Spacing & Layout (existing 8-point grid — unchanged in Phase 4)

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Minimum gaps, icon-to-text spacing |
| sm | 8px | Tab gaps, icon spacing |
| md | 12px | Component internal padding |
| lg | 16px | Card gaps, section spacing |
| xl | 24px | Card padding, feed column padding |
| 2xl | 32px | Featured card padding |
| 3xl | 48px | Section-level gaps |

Exceptions:
- TopBar height: 44px (touch-target minimum; matches existing 2.75rem definition)
- Focus ring offset uses 2px inner + 4px outer (defined in `--focus-ring` token)
- `md` (12px): 3× base-unit half-step on the 4px grid; used for component internal padding. Retained from v1.0 design system. No usages are changed in Phase 4.

**Phase 4 note:** No spacing token changes in this phase. The spacing scale is carried forward unchanged from v1.0.

## Typography

**Source:** D-05 (3-weight system), D-06 (Noto Sans JP only), D-08 (300/400/600 weights), DTKN-05

| Role | Size | Weight | Line Height | CSS Variable Binding |
|------|------|--------|-------------|---------------------|
| Meta / label | 11px (0.6875rem) | 300 (light) | 1.4 | `font-weight: 300` via `--font-jp` |
| Body / UI text | 13px (0.8125rem) | 400 (regular) | 1.5 | `font-weight: 400` via `--font-jp` |
| Card title | 15px (0.9375rem) | 600 (semibold) | 1.3 | `font-weight: 600` via `--font-jp` |
| Heading / modal | 18px (1.125rem) | 600 (semibold) | 1.2 | `font-weight: 600` via `--font-jp` |

**Font family declaration (Phase 4 output):**
```css
:root {
  --font-jp: "Noto Sans JP Variable", system-ui, sans-serif;
}
body {
  font-family: var(--font-jp);
}
```

**Font import (Phase 4 output, in `src/main.tsx`):**
```typescript
import "@fontsource-variable/noto-sans-jp/wght.css";
```

**Critical:** Font-family name is `"Noto Sans JP Variable"` (not `"Noto Sans JP"`). Name mismatch causes permanent FOIT — browser falls back to system-ui silently.

## Color

**Source:** DTKN-01 through DTKN-03, D-04, D-12, globals.css current state

### 60/30/10 Split

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--surface-base`: `#0a0a0f` (void black, locked by D-01) | Page background layer 0 |
| Secondary (30%) | Surface layers 1–4: `[STITCH-VALUE]` | Cards, panels, nav, sidebar, containers |
| Accent (10%) | `--primary`: `[STITCH-VALUE]` (purple family) + content-type accents | See "Accent reserved for" below |
| Destructive | `--error`: `#ff6e84` (carried forward) | Error states, danger button, hot badge |

### 5-Layer Surface Hierarchy (DTKN-01)

All HEX values except `--surface-base` require Stitch session confirmation (D-02).

| Token | HEX | Semantic Role |
|-------|-----|---------------|
| `--surface-base` | `#0a0a0f` | Layer 0: absolute void base (new in Phase 4; deepens from current `--surface: #0e0e13`) |
| `--surface` | `[STITCH-VALUE]` | Layer 1: page background |
| `--surface-container` | `[STITCH-VALUE]` | Layer 2: card / panel background |
| `--surface-container-high` | `[STITCH-VALUE]` | Layer 3: hovered card state |
| `--surface-container-highest` | `[STITCH-VALUE]` | Layer 4: border / divider |
| `--surface-elevated` | `[STITCH-VALUE]` | Layer 5: raised elements (deepdive drawer, secondary panel) — absorbs `--bg-secondary` and `--bg-deepdive` per D-12 |

Note: `--surface-sunken` is an alternative name for Layer 5 if visual hierarchy shows it sits below Layer 1 rather than above Layer 4. Final naming confirmed at Stitch session.

### Neon Glow System (DTKN-02)

| Token | Value | Usage (60-30-10 rule) |
|-------|-------|----------------------|
| `--glow-primary` | `rgba([R],[G],[B], 0.12)` | 60% dominant glow — primary interactive elements, active states |
| `--glow-secondary` | `rgba([R],[G],[B], 0.08)` | 30% supporting — secondary accents, nav hover |
| `--glow-subtle` | `rgba([R],[G],[B], 0.04)` | 10% ambient — background shimmer, card atmosphere |

RGB channels derive from `--primary` HEX value confirmed at Stitch session. Naming follows existing `--primary-soft` / `--primary-glow` namespace per design-system.md.

### Content-Type Accent Colors (DTKN-03)

| Token | Hue Family | Usage |
|-------|-----------|-------|
| `--accent-anime` | Purple (same family as `--primary`) | Anime content cards — left border + badge background (Phase 5) |
| `--accent-manga` | Pink / Magenta | Manga content cards — left border + badge background (Phase 5) |
| `--accent-game` | Cyan | Game content cards — left border + badge background (Phase 5) |
| `--accent-news` | Amber / Gold | News content cards — left border + badge background (Phase 5) |

**WCAG AA requirement:** Each accent MUST pass 4.5:1 contrast against `--surface-container` when used as text. Cyan (`--accent-game`) and amber (`--accent-news`) are highest-risk — verify each after Stitch session using webaim.org/resources/contrastchecker. If below 4.5:1, increase lightness until compliant.

**Phase 4 scope:** CSS variable definitions only. No components are updated in Phase 4. Phase 5 applies `--accent-*` to card left borders and badge backgrounds.

### Accent Reserved For

The 10% accent budget (`--primary`, `--accent-anime`, `--accent-manga`, `--accent-game`, `--accent-news`) is reserved for:
- Primary CTA buttons (`--primary`)
- Active/selected navigation item indicator (`--primary`)
- Active tab indicator (`--primary`)
- Focus ring outer glow (`--primary`)
- Content-type card left border (Phase 5, using specific `--accent-*`)
- Content-type badge background (Phase 5, using specific `--accent-*`)

Accent is NOT applied to: body text, card backgrounds, dividers, placeholder text, or inactive UI elements.

## Copywriting Contract

**Source:** REQUIREMENTS.md success criteria, ROADMAP.md Phase 4 success criteria

Phase 4 is a token/CSS layer with no UI-visible user interaction surfaces introduced or changed. However, the following states must remain functional throughout (regression protection):

| Element | Copy | Notes |
|---------|------|-------|
| Primary CTA | "Save Settings" (ProfileWing) / "Refresh" (ScheduleWing) | Existing CTAs unchanged — no new CTAs introduced in Phase 4 |
| Empty state heading | "Nothing here yet" | Existing copy — Skeleton UI handles loading state per design.md §6 |
| Empty state body | "No articles found. Try refreshing your feeds." | Existing copy — carried forward |
| API error toast | "Failed to load. Check your connection and try again." | Existing error pattern via `showToast('error', message)` |
| Font load failure | System font fallback renders silently — no user-facing error copy | `font-display: swap` ensures system font shows immediately; no error state needed |
| Biome check failure (dev) | "npm run check failed — fix lint errors before committing." | Internal dev copy, not user-facing |

**Destructive actions in Phase 4:** None. The Legacy alias removal (DTKN-06) is a code operation with no user-facing confirmation dialog. The only destructive action in scope is deleting the Legacy aliases CSS block, which requires the grep gate to return zero matches — enforced programmatically, not via user confirmation.

## Interaction Contract

Phase 4 introduces no new interaction surfaces. All existing interactions (hover states, focus rings, active states) continue to function via inherited tokens. No new animations, transitions, or motion are added.

**Hover depth feedback:** Currently implemented via `--surface-container-high` background + `--outline-variant` border + `--shadow-md`. These tokens are preserved through the alias migration. No behavior change.

**Focus ring:** `--focus-ring` token preserved unchanged. All keyboard navigation continues to work identically.

**Reduced motion:** `@media (prefers-reduced-motion: reduce)` block in `globals.css` preserved unchanged.
