<objective>
Implement graceful scheduler shutdown (RESL-01) and settings hot-reload (RESL-02).

Purpose: The scheduler's collect_loop and digest_loop currently run indefinitely with no clean shutdown path, risking DB corruption on window close. Settings changes require app restart. This plan adds CancellationToken-based shutdown with a 5-second timeout, and watch-channel-based config hot-reload.

Output: Modified scheduler.rs with select!-based cancellation, lib.rs with shutdown hook and watch channel, commands/scheduler.rs with watch::Sender integration.
</objective>

**Cargo.toml changes:**
Add explicit dependency: `tokio-util = { version = "0.7", features = ["rt"] }` and `url = "2.5"` (url is for Plan 03 but add now to avoid duplicate edits).

**lib.rs changes (per D-01, D-04, D-05):**
1. Import `tokio_util::sync::CancellationToken` and `tokio::sync::watch`
2. In `run_setup()` (or equivalent setup function):
   - Create `let token = CancellationToken::new();`
   - Create `let (config_tx, config_rx) = tokio::sync::watch::channel(initial_scheduler_config);` where `initial_scheduler_config` is loaded from DB settings (existing pattern)
   - `app.manage(token.clone());` — registers as `State<CancellationToken>` (per D-01: independent, NOT inside AppState)
   - `app.manage(Arc::new(config_tx));` — registers as `State<Arc<watch::Sender<SchedulerConfig>>>`
   - Pass `token.clone()` and `config_rx` to `scheduler::start()`
3. Add `.on_window_event(|window, event| { ... })` BEFORE `.invoke_handler(...)`:
   - Match `tauri::WindowEvent::CloseRequested { api, .. }`
   - Call `api.prevent_close();`
   - Clone `window.app_handle()`
   - `tauri::async_runtime::spawn(async move { ... })`:
     - Get token: `if let Some(token) = handle.try_state::<CancellationToken>() { token.cancel(); }`
     - Wait: `let _ = tokio::time::timeout(std::time::Duration::from_secs(5), async { tokio::time::sleep(std::time::Duration::from_secs(5)).await }).await;`
     - NOTE: Since JoinHandles are hard to store in State (not Send+Sync), rely on the 5-second timeout as a fixed grace period. The loops break on CancellationToken nearly instantly; the timeout is the safety net.
     - Call `handle.exit(0);`

**scheduler.rs changes (per D-02, D-03, D-04):**
1. Change `start()` signature to accept `token: CancellationToken` and `config_rx: tokio::sync::watch::Receiver<SchedulerConfig>` as additional parameters
2. Pass `token.clone()` and `config_rx.clone()` to both `collect_loop()` and `digest_loop()` spawn calls
3. In `collect_loop()`:
   - Add `token: CancellationToken` and `mut config_rx: watch::Receiver<SchedulerConfig>` parameters
   - Replace the existing `loop { ... tokio::time::sleep(...).await ... }` with:
   ```rust
   let mut timer = tokio::time::interval_at(
       tokio::time::Instant::now() + Duration::from_secs(60 * config.collect_interval_minutes as u64),
       Duration::from_secs(60 * config.collect_interval_minutes as u64),
   );
   loop {
       tokio::select! {
           _ = token.cancelled() => {
               tracing::info!("collect_loop: shutdown signal received");
               break;
           }
           _ = config_rx.changed() => {
               let new_config = config_rx.borrow_and_update().clone();
               // Rebuild timer with new interval
               let dur = Duration::from_secs(60 * new_config.collect_interval_minutes as u64);
               timer = tokio::time::interval_at(tokio::time::Instant::now() + dur, dur);
               tracing::info!(interval_min = new_config.collect_interval_minutes, "collect_loop: config updated");
           }
           _ = timer.tick() => {
               // existing collect logic (check enabled flag from config_rx.borrow())
               let config = config_rx.borrow().clone();
               if !config.enabled { continue; }
               // ... existing refresh_all() call ...
           }
       }
   }
   ```
4. Apply the same `select!` pattern to `digest_loop()`:
   - CancellationToken branch for shutdown
   - config_rx.changed() branch for digest_hour/digest_minute updates
   - Existing sleep-until-digest-time logic in the timer branch
   - Check `config.enabled` before generating digest

**commands/scheduler.rs changes (per D-06, D-07):**
1. Add `tx: State<'_, Arc<watch::Sender<SchedulerConfig>>>` parameter to `set_scheduler_config()`
2. After saving config to DB (existing logic), send to watch channel:
   ```rust
   tx.send(new_config.into()).map_err(|e| AppError::Internal(format!("Failed to broadcast config: {e}")))?;
   ```
3. Also emit Tauri event for frontend notification (per D-07):
   ```rust
   app_handle.emit("scheduler-config-changed", &new_config)?;
   ```

**state.rs: No changes needed.** CancellationToken and watch channel are managed independently via `app.manage()`, not inside AppState (per D-01, state_no_mutex.md pattern).
