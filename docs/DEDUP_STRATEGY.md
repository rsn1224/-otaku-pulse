# 重複排除（Dedup）戦略設計書

<!-- 最終更新: 2026-03-19 -->

> OtakuPulse における記事の重複排除アルゴリズムを定義する。
> 3 層構造で段階的に重複を検出し、高速かつ正確な排除を実現する。

---

## 1. 3 層重複排除アルゴリズム概要

```text
[新規記事取得]
    │
    ├─ Layer 1: URL 正規化（完全一致） → 高速・高信頼
    │      一致 → UPSERT（既存記事を更新）
    │
    ├─ Layer 2: タイトル Jaccard bigram 類似度（近似） → 中速・主エンジン
    │      類似度 >= 0.6 → 重複候補としてマーク
    │
    └─ Layer 3: コンテンツハッシュ（完全一致） → 低速・最終手段
           一致 → 重複としてマーク
```

| Layer | 手法 | 速度 | 精度 | 役割 |
|-------|------|------|------|------|
| 1 | URL 正規化 + 完全一致 | ★★★ | ★★★ | 同一記事の更新検出 |
| 2 | タイトル Jaccard bigram | ★★☆ | ★★☆ | 異なるソースからの同一ニュース検出 |
| 3 | コンテンツハッシュ | ★☆☆ | ★★★ | タイトルが異なるが本文同一の記事検出 |

---

## 2. Layer 1: URL 正規化

### 正規化ルール

1. **トラッキングパラメータの除去**: `utm_*`, `ref`, `source`, `fbclid`, `gclid`
2. **末尾スラッシュの統一**: `/path/` → `/path`
3. **プロトコルの統一**: `http://` → `https://`
4. **www の統一**: `www.example.com` → `example.com`
5. **フラグメントの除去**: `#section` を削除
6. **クエリパラメータのソート**: 順序の違いによる不一致を防止

### 正規化の例

```text
入力: http://www.example.com/article/123?utm_source=twitter&ref=top#comment
出力: https://example.com/article/123
```

### ストレージ

- 正規化済み URL は `articles.url_normalized` カラムに保存
- `UNIQUE` 制約を設定し、DB レベルで完全一致を保証
- UPSERT 時に既存記事の `updated_at` を更新し、`is_read` は保持

### 除去対象パラメータ一覧

```text
utm_source, utm_medium, utm_campaign, utm_term, utm_content,
ref, source, fbclid, gclid, msclkid, mc_cid, mc_eid
```

## 3. Layer 2: タイトル Jaccard bigram 類似度

### なぜ bigram か

日本語タイトルの類似度計算において、形態素解析（MeCab 等）は外部依存が大きく、ビルド・配布の複雑さが増す。**bigram（2 文字スライディングウィンドウ）** は以下の利点がある:

- 外部ライブラリ不要（pure Rust で実装可能）
- 日本語・英語の混在タイトルに対応
- 計算コストが低い
- ニュースタイトルの類似度検出には十分な精度

### アルゴリズム

```text
Jaccard 係数 = |A ∩ B| / |A ∪ B|

A = タイトル A の bigram 集合
B = タイトル B の bigram 集合
```

### 例

```text
タイトル A: "進撃の巨人 最終回"
bigrams A: {"進撃", "撃の", "の巨", "巨人", "人 ", " 最", "最終", "終回"}

タイトル B: "「進撃の巨人」最終回放送決定"
bigrams B: {"「進", "進撃", "撃の", "の巨", "巨人", "人」", "」最", "最終", "終回", "回放", "放送", "送決", "決定"}

共通: {"進撃", "撃の", "の巨", "巨人", "最終", "終回"} = 6
和集合: 15
Jaccard = 6 / 15 = 0.40

※ 記号除去の前処理後に計算するため、実際のスコアはより高くなる
```

### 閾値

| Jaccard スコア | 判定 |
|---------------|------|
| >= 0.6 | 重複候補（`is_duplicate = true`） |
| 0.4 - 0.59 | 関連記事（将来の「関連記事」機能用に記録） |
| < 0.4 | 別記事 |

3. **小文字化**: 英字を小文字に統一
4. **空白の正規化**: 連続空白を単一スペースに

## 4. Layer 3: コンテンツハッシュ

### 方式

- **ハッシュ関数**: SHA-256
- **入力**: 記事本文の先頭 200 文字（前処理後）
- **保存先**: `articles.content_hash` カラム（専用カラム + `idx_articles_content_hash` インデックス）

### なぜ先頭 200 文字か

- 記事全文のハッシュは更新時に変化するため不安定
- 先頭 200 文字はリード文（要旨）を含み、記事の同一性判定に十分
- 計算コストを最小化

### ハッシュ生成手順

```text
1. 記事本文を取得
2. HTML タグを除去
3. 空白を正規化
4. 先頭 200 文字を切り出し
5. SHA-256 ハッシュを計算
6. 16進文字列として保存
```

### 判定

- **完全一致のみ**: ハッシュが一致した場合のみ重複と判定
- 近似一致は行わない（Layer 2 の役割）

## 5. 処理ルールと UI 連携

### UPSERT ルール（Layer 1 一致時）

```text
URL 完全一致 → 既存記事を更新
  - title: 新しい値で上書き
  - body: 新しい値で上書き
  - updated_at: 現在時刻に更新
  - is_read: 既存値を保持（ユーザー状態は維持）
  - importance_score: 再計算（要約がある場合）
```

### 重複マークルール（Layer 2/3 一致時）

```text
重複検出 → 新規記事に以下を設定
  - is_duplicate: true
  - duplicate_of: {元記事の ID}
  - 元記事の選定基準: importance_score が高い方を代表記事とする
```

### UI での扱い

| 状態 | 表示 |
|------|------|
| `is_duplicate = false` | 通常表示 |
| `is_duplicate = true` | デフォルト非表示（フィルターで表示可能） |
| 重複グループ | 代表記事に「他 N 件」バッジ表示 |

### 代表記事の選定

同一ニュースが複数ソースから取得された場合:

1. `importance_score` が最も高い記事を代表とする
2. スコアが同じ場合は `published_at` が最も早い記事を代表とする
3. UI では代表記事のみ表示し、展開すると全ソースを一覧できる

## 6. パフォーマンス考慮

| 処理 | 想定時間 | 備考 |
|------|---------|------|
| URL 正規化 + DB 検索 | < 1ms/記事 | UNIQUE インデックス活用 |
| Jaccard bigram 計算 | < 1ms/ペア | 同一カテゴリ内の直近 100 記事と比較 |
| SHA-256 ハッシュ計算 | < 0.1ms/記事 | 固定長入力のため高速 |

### 比較範囲の制限

- Layer 2 の比較は**同一カテゴリ内の直近 7 日間の記事**に限定
- これにより比較ペア数を抑制し、O(n^2) の爆発を防ぐ
- 7 日を超えた記事との重複は実運用上ほぼ発生しない
