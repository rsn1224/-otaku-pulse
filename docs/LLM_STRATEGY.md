# LLM 戦略設計書

<!-- 最終更新: 2026-03-19 -->

> OtakuPulse における LLM 活用方針を定義する。
> ローカル LLM（Ollama）優先、フォールバックとして OpenAI API、最終手段として要約スキップ。

---

## 1. 推奨モデル一覧（2026-03 時点）

| 優先度 | モデル | VRAM 目安 | 日本語性能 | Ollama コマンド |
|--------|--------|-----------|-----------|----------------|
| 1（推奨） | Qwen 2.5 7B Instruct | ~5-6 GB（Q4_K_M） | ★★★ | `ollama pull qwen2.5:7b-instruct` |
| 2 | Gemma 3 12B | ~12 GB（Q4_K_M） | ★★★ | `ollama pull gemma3:12b` |
| 3 | Qwen 3 8B | ~6-7 GB（Q4_K_M） | ★★★ | `ollama pull qwen3:8b` |
| FB（フォールバック） | gpt-4.1-mini（API） | - | ★★★ | OpenAI API |

### 選定理由

- **Qwen 2.5 7B Instruct**: VRAM 6GB 以下で動作し、日本語要約品質が最も安定。RTX 3060/4060 クラスで快適に推論可能。Instruct チューニング済みのためプロンプト追従性が高い。
- **Gemma 3 12B**: 品質は最高レベルだが VRAM 12GB 必要。RTX 4070 以上推奨。
- **Qwen 3 8B**: Qwen 2.5 の後継。性能向上しているが、thinking モード（推論モード）がデフォルト有効のため、要約用途では明示的に無効化が必要。
- **gpt-4.1-mini**: Ollama が利用不可の場合のフォールバック。API キーが必要。コスト: ~$0.40/1M input tokens。

## 2. Ollama API 仕様

### エンドポイント

```
POST http://localhost:11434/api/generate
```

### リクエストボディ

```json
{
  "model": "qwen2.5:7b-instruct",
  "prompt": "<後述のプロンプトテンプレート>",
  "stream": false,
  "options": {
    "temperature": 0.3,
    "top_p": 0.9,
    "num_predict": 512
  }
}
```

### パラメータ説明

| パラメータ | 値 | 理由 |
|-----------|-----|------|
| `temperature` | 0.3 | 要約は創造性不要。低めで安定出力を優先 |
| `top_p` | 0.9 | 極端な出力を抑制しつつ自然な文章を維持 |
| `num_predict` | 512 | 要約は短文。トークン制限で冗長出力を防止 |
| `stream` | false | バッチ処理のため一括レスポンスが効率的 |

### Qwen 3 固有の設定

Qwen 3 は thinking モード（内部推論）がデフォルト有効。要約タスクには不要なため明示的に無効化する。

```json
{
  "model": "qwen3:8b",
  "prompt": "<プロンプト>",
  "stream": false,
  "think": false,
  "options": {
    "temperature": 0.3,
    "top_p": 0.9,
    "num_predict": 512
  }
}
```

> **注意**: `think: false` を設定しないと推論トークンが大量に生成され、レスポンス時間が 3-5 倍になる。

## 3. フォールバック戦略

```text
[ヘルスチェック] GET /api/tags
       │
       ├─ 200 OK → Ollama ローカル LLM で要約
       │
       └─ 接続失敗 / タイムアウト
              │
              ├─ OPENAI_API_KEY あり → gpt-4.1-mini で要約
              │
              └─ API キーなし → 要約スキップ（本文先頭 200 文字を表示）
```

### ヘルスチェック詳細

- **エンドポイント**: `GET http://localhost:11434/api/tags`
- **タイムアウト**: 3 秒
- **判定**: HTTP 200 かつレスポンスに設定済みモデル名が含まれること
- **リトライ**: なし（フォールバックに即座に切り替え）

### OpenAI フォールバック

- **モデル**: `gpt-4.1-mini`
- **API キー**: 環境変数 `OPENAI_API_KEY` または設定ファイルから取得
- **タイムアウト**: 30 秒
- **レート制限**: 1 リクエスト/秒（バッチ処理時）

### 要約スキップ時の表示

```text
{article.body の先頭 200 文字}...
```

UI 上では「AI 要約: 利用不可」のバッジを表示し、手動再生成ボタンを提供する。

## 4. カテゴリ別要約プロンプトテンプレート

### 共通ルール

- **システムプロンプト**: 日本語で記述
- **出力形式**: タイトル / 2-3 文の要約 / 重要度スコア（0.0-1.0）
- **入力制限**: 記事本文は先頭 2,000 文字に切り詰め
- **バッチ制限**: 1 プロンプトあたり最大 10 記事/カテゴリ

### アニメカテゴリ

```text
あなたはアニメニュースの要約アシスタントです。
以下の記事を読み、JSON形式で要約してください。

出力形式:
{"title": "記事タイトル", "summary": "2-3文の要約", "importance_score": 0.0-1.0}

重要度の基準:
- 0.8-1.0: 新作発表、放送開始、制作会社の重大ニュース
- 0.5-0.7: キャスト発表、PV公開、イベント情報
- 0.0-0.4: 感想まとめ、ランキング、コラム

記事:
{article_body}
```

### マンガカテゴリ

```text
あなたはマンガニュースの要約アシスタントです。
以下の記事を読み、JSON形式で要約してください。

出力形式:

重要度の基準:
- 0.8-1.0: 新連載開始、連載終了、アニメ化決定
- 0.5-0.7: 新刊発売、作家インタビュー、受賞
- 0.0-0.4: 感想、ランキング、コラム

記事:
{article_body}
```

### ゲームカテゴリ

```text

- 0.8-1.0: 新製品発表、ベンチマーク結果、重大な不具合報告
```

### バッチプロンプト（複数記事）

```text

```

## 5. バッチ処理 vs リアルタイム処理

| モード | 用途 | タイミング | 備考 |
|--------|------|-----------|------|
| バッチ | ダイジェスト生成 | 毎日 AM 7:00 JST | 全カテゴリの未要約記事を一括処理 |
| リアルタイム | 重要記事の即時要約 | `importance_score >= 0.7` | フィード取得時にタイトルで事前スコアリング |

### バッチ処理フロー

```text
[cron: 07:00 JST] → 未要約記事を取得
    → カテゴリ別に分類
    → 10 記事ずつバッチプロンプト生成
    → LLM に送信（フォールバック付き）
    → 要約結果を DB に保存
    → ダイジェスト画面用データを更新
```

### リアルタイム処理フロー

```text
[フィード取得] → 新規記事を検出
    → タイトルベースの事前スコアリング（キーワードマッチ）
    → スコア >= 0.7 の記事のみ即時要約
    → 通知トリガー（設定が有効な場合）
```

### 事前スコアリングキーワード（例）

| カテゴリ | 高重要度キーワード |
|----------|-------------------|
| アニメ | 新作, 放送開始, 制作決定, アニメ化 |
| マンガ | 新連載, 最終回, アニメ化決定 |
| ゲーム | 新作発表, 発売日, ローンチ |
| PC | 発売, ベンチマーク, リコール, 不具合 |

## 6. パフォーマンス目標

| 指標 | 目標値 |
|------|--------|
| 単一記事の要約レスポンス時間（ローカル） | < 5 秒 |
| バッチ 10 記事の処理時間（ローカル） | < 30 秒 |
| OpenAI フォールバック時のレスポンス時間 | < 10 秒 |
| 1 日あたりの LLM 呼び出し回数 | ~50-100 回 |
